# ~60 fps motion (optional, VapourSynth)

**Name:** Optional path to about **60 fps** display cadence for film-like sources using **synthesized in-between** frames (not **mpv** display-resample; that path is **removed** — see [25-smooth-playback](25-smooth-playback.md)).

**Implementation status:** Done (SQLite `video_smooth_60`, `video_vs_path`, `video_mvtools_lib` cache for `libmvtools.so`; main **Preferences** → checkable **Smooth Video (~60 FPS at 1.0×)** — stores user **intent**; the bundled [vf] runs at **~1.0×** only; single bundled [`data/vs/rhino_60_mvtools.vpy`](../data/vs/rhino_60_mvtools.vpy) when `video_vs_path` is empty — **FlowFPS** using the full detected CPU affinity + deeper cache/queue, `LoadPlugin` + **`RHINO_MVTOOLS_LIB`**; custom `.vpy` path optional; mvtools + VapourSynth + mpv VS build required).

**Use cases:** Viewers who want more **temporal** smoothness (often called “soap opera” or HFR) on a **~60 Hz** display.

**Short description:** A user-supplied [`.vpy` script](https://vapoursynth.com/doc) using `core.std` and plugins from [vsrepo] (e.g. **MVTools** for motion, or **RIFE-ncnn** / VSGAN for ML quality). The app ships one **MVTools FlowFPS** ~60p baseline when no custom path is set.

**Long description:** The **VapourSynth** community uses filters such as [MVTools] block motion, [RIFE] / ncnn, or DAIN exports for fluid motion; there is no single “official” 60p preset—**quality, speed, and GPU use depend on the script and plugins**. The **mpv** [vapoursynth] filter runs a **Python** script with a global `video_in` clip; a separate mpv build may omit the filter, so the app only adds the vf when the feature exists (otherwise stderr notes). **Integrated** graphics: VapourSynth + ncnn/ML filters may use the GPU in some builds, but **MVTools** is **CPU**-oriented; lower resolution or a stronger CPU helps.

**Specification:**

- **Settings** (SQLite `settings`): `video_smooth_60` = `0` / `1`; `video_vs_path` = UTF-8 path to `.vpy` (may be empty for bundled script); `video_mvtools_lib` = cached absolute path to `libmvtools.so` after a successful find (reused on next run if the file still exists, so the app does not rescan). [save/load](14-preferences.md) with other video prefs.
- **Menu:** **Preferences → Smooth Video (~60 FPS at 1.0×)** (stateful bool GAction `smooth-60`); the label reflects *when* the mvtools path runs, not a separate extra toggle. **Changing playback speed** does **not** uncheck the item — prefs are unchanged; only the **vapoursynth** [vf] is skipped when [mpv] `speed` is not ~**1.0×**. When `video_vs_path` is set, a second row shows the **script file name** (basename only) with a stateful GAction `vs-custom` (checkbox) — unchecking **clears** `video_vs_path` (reverts to the bundled `.vpy`) and reapplies `vf`. **Choose VapourSynth Script…** sets the path, turns Smooth 60 on, and saves only after `libmvtools.so` is found. With no player, prefs + menu update; if `libmvtools.so` is not found, Rhino keeps Smooth 60 off and opens a setup dialog. With a player, `vf` is reapplied only after MVTools is resolved. If Smooth 60 is auto-disabled because `libmvtools.so` cannot be resolved, the app opens the same modal setup dialog with selectable copy-paste commands for installing VapourSynth, `vsrepo`, MVTools, and a VapourSynth-capable mpv build. The dialog also appears after choosing/loading a `.vpy` when `libmvtools.so` is missing, and the chosen path is not saved in that case. The **Preferences** submenu is rebuilt when the custom path is added or removed so the name row appears or disappears.
- **Playback speed & bundled script (with [28](28-playback-speed.md)):** The app only installs the `vapoursynth` [vf] when [mpv] **speed** is **~1.0×**; at **1.5×/2.0** (or any other rate) the [vf] is omitted and decode stays on the **auto** `hwdec` path—**Smooth 60** is not auto-unchecked; after a header **speed** change, the app defers [apply_mpv_video] to the next **GLib idle** so **libmpv** has applied [speed] before the [vf] graph is rebuilt, and passes the row value as a hint so **1.0×** is not missed. Before the [vf] is added, the app sets **`RHINO_PLAYBACK_SPEED`** in the process environment to the current [mpv](https://mpv.io/manual/master/#options-speed) `speed` (e.g. `1.0`, `1.5`, `2.0`). The bundled `rhino_60_mvtools.vpy` uses **exact** rationals for those speeds. At **1.0** it only tags the source with **`AssumeFPS`** once, then **FlowFPS** to **60/1** (no second **`AssumeFPS(×1)`**), so timing matches the pre–speed-ramp script and does not skew A/V. At **1.5** / **2.0** it **also** retimes the clip with **`AssumeFPS(× speed)`** before **FlowFPS**, so **(source fps × speed)** is the content rate mvtools sees and only the “missing” in-betweens toward 60 are synthesized. If **(source fps as `Fraction` × speed)** is **≥ 60/1** in **rational** form, the script **skips** mvtools (faster; e.g. exact 30×2). A **float + ε** threshold was **removed**: tags like **60000/1001** (≈59.94) must **not** skip **FlowFPS**, or real **24p** often looked un-smoothed. When the user changes `speed` in the UI, **vf clr** and **re-add** run so the script reloads; the app writes **`RHINO_PLAYBACK_SPEED` from the row value** before rebuild to avoid a same-tick **[get_property](https://mpv.io/manual/master/#command-interface)** race with [mpv] `set_property`. A **~320 ms** one-shot after [loadfile] (on-file-loaded) resyncs **Smooth 60** / `vf` / env so the header list **snaps** [mpv] `speed` to 1.0/1.5/2.0, **`RHINO_PLAYBACK_SPEED` matches** [mpv] when [watch-later](https://mpv.io/manual/master/#options-save-position-on-quit) restored `speed`, and the [vf] string matches the **~1.0×** mvtools rule (e.g. no script at sped-up rate) — all **event-driven**, not the transport **~200 ms** tick. **Custom** `.vpy` files can read **`RHINO_PLAYBACK_SPEED`** the same way or ignore it.
- **mpv:** normal playback is left on mpv defaults: no forced `video-sync`, `interpolation`, `hwdec`, `vd-lavc-dr`, or `vf clr`. The VapourSynth graph itself creates the ~60 fps frames; GTK’s frame clock and libmpv `report_swap` remain the presentation/vsync path. The app adds `vapoursynth:file=…:buffered-frames=24:concurrent-frames=auto` only when the toggle is on, media is open, and `speed` is **~1.0×**. The deeper VapourSynth queue gives mvtools more room to keep frames ready, improving smoothness at the cost of a little extra memory and startup/seek latency. On paused seek while `vapoursynth` is active, the app temporarily clears the **vf** before seeking so mpv can render a normal still frame instead of leaving the `GLArea` black; Smooth 60 is reapplied on the next Play/Space unpause if the preference is still enabled. The bundled script uses all detected CPU threads in the process CPU affinity for MVTools, and keeps the original fast `pel=1`, 32px block, `search=0`, `FlowFPS(mask=0)` graph, adds 16px block overlap so adjacent blocks blend at their edges, and enables only `global=True` for broad camera pans; stronger `truemotion` / low `pglobal` bias is intentionally avoided because it can look like vertical tearing on mixed-motion scenes. While that [vf] is being installed, **`hwdec` is forced to `no`** and **`vd-lavc-dr` to `no`** (direct rendering can also bypass the CPU filter path). The default `hwdec=auto` (VAAPI, NVDEC, etc.) often **skips the CPU VapourSynth path** — the classic “it does nothing” report. When an existing VapourSynth graph must be removed (toggle off, rejected vf, or **speed is not ~1.0×**), the app clears the `vf` list and restores **`hwdec` / `vd-lavc-dr` to `auto`**. If no VapourSynth graph is active, those settings are not touched. **Reapply after [loadfile]:** [try_load] runs [apply_mpv_video] on the first **GLib idle**, then chains a **second idle** with the *reapply 60 if still missing* path if the `vapoursynth` line is still missing while **Smooth 60** is on and [speed] is **~1.0×** (post-load graph race; **no** wall-clock timer). A **brief black** is normal while the graph starts; the app does **not** add VapourSynth in a second delayed **loadfile** step. **After** an actual [vf] clear/replace, the code runs **`seek <time-pos> absolute+keyframes`** to re-align A/V; if `time-pos` < ~0.12s it is skipped to avoid fighting [try_load]. If **seek** fails, **`time-pos`** is set. Same parallel `getFrameAsync` notes; `vd-lavc-threads=0` (auto) at init. If the `vapoursynth` **vf** cannot be added **while the toggle is on and speed is ~1.0×** (mvtools should run), the app sets **`video_smooth_60` to `0`**, saves, and unchecks the menu (see `data/vs/README.md`, `scripts/ensure-vapoursynth-debian.sh`). If [vf] `add` fails, the code tries [set_property] `vf` as a fallback. The app does **not** run a periodic poll on `vf` to detect a filter that mpv **removed after** a successful add; fix mvtools / the script, or turn **Smooth 60** off. **mvtools path:** when adding the `vf`, the app resolves `libmvtools.so` in order: **`RHINO_MVTOOLS_LIB`** if set; else **cached** `video_mvtools_lib` if that path is still a file (skips disk search); else search (common distro locations, pipx/vsrepo under `~/.local`, bounded search of the rest of `~/.local`). On success, saves the absolute path to **`video_mvtools_lib`**. Stderr: `libmvtools -> …` (including `cached in settings`); bundled `.vpy` use `LoadPlugin` with the same fallbacks. Custom script paths are unchanged.
- **A/B testing:** **Subtitles** (libass) are not inside the VapourSynth graph; a scrolling line can look similar at 24p vs 60p. Check **video motion** (pans) to judge HFR, or turn subs off for a second.
- **Default** when the DB has no relevant keys: toggle **on** (bundled **FlowFPS** `data/vs/rhino_60_mvtools.vpy`, full-affinity multithreading + moderate cache). 1080p is still **CPU**-bound.
