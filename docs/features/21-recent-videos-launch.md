# Recent videos grid on empty launch

**Name:** Recent videos grid (no CLI / no open-on-start)

**Implementation status:** Done (no CLI: recent grid, **`~/.config/rhino/rhino.sqlite`**, **raster** thumb BLOBs, libmpv `vo=image` + `screenshot-to-file` pipeline; **session** LIFO **undo** stack for dismiss; session-restore TBD for “who wins” with [16](16-session-persistence.md))

**Use cases:** Users who launch the app from the icon get back to what they were watching without digging through the file manager; at a glance they see up to five recent files, how far they got, and can resume with one click.

**Short description:** When the app starts with **no files or URLs from the command line** and **no other “open this first” path** (e.g. session restore) takes over the first paint, the main area shows a **row or grid of five** recent items. Each item shows a **thumbnail**, the **file name** (not the directory path: full name, word-wrapped on the card as needed), a **progress bar** with **percent complete**, and opens the file when the user **clicks the thumbnail (or the whole card)**. Order is **most recently opened first** (backwards **history** order).

**Long description:** This view is the **default empty state** of the main content where video normally renders (not a separate dialog). History is a **durable list** of opened local files (and optionally streams later; see [Open files and CLI](06-open-and-cli.md)) maintained whenever the user successfully opens a path from any supported entry point. Only **up to five** entries are **shown**; the backing list may keep more for future use. **Thumbnails** are generated or cached in a well-defined way (e.g. under the user cache directory) so repeat opens are fast; details may align with [Thumbnail seek-bar preview](18-thumbnail-preview.md) to avoid two unrelated pipelines. **Progress** uses the same resume story as [Preferences and persistent settings](14-preferences.md) / watch-later: e.g. `time-pos` and `duration` to compute a **percentage**; if duration is unknown, show an **indeterminate** bar or **0%** and document the choice. If there are **fewer than five** history entries, show only the available slots (no empty placeholders unless we explicitly want them for layout—default: no placeholders). If **history is empty**, show a short **“Open a file”** affordance (consistent with the status line) or the same empty hint already used in the app. **Opening a file** from a card uses the same **load** path as the file picker (canonical path, resume). Navigating back from playback to this grid is out of scope for v1 unless specified later.

**Specification:**

- **Trigger:** The grid appears when the first window is shown and **no CLI arguments** request a path, and the app is not (for this release) **replacing** that with a mandatory “restore last session” full-playlist load that fills the main view—see [Session: restore last playlist](16-session-persistence.md) for coordination; the spec for “who wins” must be a single line in this file when session restore ships.
- **Play on card open:** A click on a recent card (or the same **Open** dialog path) must **unpause** after `loadfile` so playback **starts** even if watch_later had stored a **paused** session for that file. Opening from the **CLI** on startup may still respect saved pause (implementation: `try_load(..., play_on_start)` in `app.rs`).
- **Count:** **Exactly up to five** visible cards, **most recently opened first**.
- **Per card UI:** (1) **Thumbnail** fills the card background at a **fixed** card size; image uses **cover** style (**crop** edges as needed, **no** letterboxing in the art). (2) **Primary label** and (3) **Progress bar** with **numeric percentage** (e.g. `42%`) sit in a **footer strip** at the bottom of the card (0% if never started, 100% or “complete” if finished per stored state; TBD: treat near-end as 100% if within N seconds). The footer has a **semi-opaque** dark background and light label text for **contrast** on bright frames. **File name** is the last path segment only, shown in full (word-wrapped, no ellipsize). Full path in tooltip optional. (4) **Click target:** the **whole card** **opens** that media; same loading and resume rules as [Open files](06-open-and-cli.md). (5) **Remove:** a small **circular** close control appears **on hover** (top-right); one click **removes** that entry from continue history with **no confirmation**; an **Undo** control appears above the card row. Each removal is pushed on an in-memory **LIFO stack** for the app run; **Undo** pops one and restores via `history::record` (same as re-open). The stack is cleared when leaving this view (Escape to browse).
- **History persistence:** Store an ordered, de-duplicated list of opens under XDG config (e.g. `~/.config/rhino/`, filename TBD). **Deduplication:** opening the same path again **moves** it to the front. **Prune:** optional max list length (e.g. 20) with oldest dropped; the UI still shows at most five. **Invalid paths** (file deleted): show a **stale** state (greyed card and/or “unavailable”); click may remove the entry or show a toast (prefer removal from list on click or a small clear action; keep UX minimal per product constraints). **Completed items:** if the user (or the sibling-folder queue) successfully loads a **different** file while the current local file is **fully played** (libmpv `eof-reached` or `time-pos` within ~3s of a known `duration`, matching near-end in the recent grid), **remove** the previous file from the continue history so the list stays a “resume / continue” list, not a log of every finished file.
- **Thumbnails:** **Raster** **BLOBs** in SQLite (`media` table; column name `thumb_png` is historical). **Grid rebuild:** if missing or mtime (whole seconds) changed, a **one-shot** libmpv with `vo=image` decodes a frame at **2 s** with `vf=scale=iw/6:…:flags=bilinear` and **JPEG** output (`vo-image-jpeg-quality` high, e.g. ~85, `vo-image-png-compression=0` if PNG)—stored images are about **1/6 linear** size vs the source frame (higher quality on the card than a fixed 200px cap). **UI thread:** the grid is painted from **DB-only** cache reads; **libmpv** thumb generation for missing BLOBs runs on a **background** thread, then the grid is refreshed on the main loop (no sequential 30s blocks on the UI thread for up to five files), using a main-thread **channel** for refill signals. When the recent scroller is destroyed, **RecentContext** is **dropped** and workers are **canceled and joined** (no intended leak of the context). **While playing / quit:** `screenshot-to-file` captures a **full-res** frame; the bytes are then **rescaled in-process** to fit within **960×540** at most (aspect ratio preserved, no upscale; Lanczos, JPEG Q ~**90**) before storing, except when a thumbnail already in the DB is for the **same file mtime** and the same [time-pos] within **0.5 s** (no redundant screenshot). Distinct from the **grid** `vo=image` path (which still uses `scale=iw/6` for small card art). **Back to recent (e.g. Escape):** the quit **screenshot/DB** step runs **before** the recent grid is rebuilt so card art reflects the just-played state. On first run, legacy `recent_files.txt` / `durations.txt` are imported and renamed to `*.migrated`.
- **Length:** The libmpv **`duration` property** is written to the DB on **file switch** and **window close** (not `ffprobe`).
- **Progress source:** On the grid, prefer **stored** `time_pos_sec` + `duration_sec` from SQLite (written on file switch / close from libmpv `time-pos` / `duration`). If `time-pos` was not stored, fall back to watch_later (`start=` / `# path`). libmpv still uses the **watch_later** directory for actual resume on open. If both are missing, show **0%**.
- **Accessibility:** Each card is a **single** activatable control or a button with a label that includes the file name; progress is exposed to screen readers in a simple form (e.g. “45 percent played”).
- **Acceptance (manual):** With history containing ≥3 local files, launch with **no** args: three cards appear in correct order, percentages match reopen behavior, click loads and seeks. With empty history, no crash and **clear** empty state. **With** a CLI file, **this grid is not** the first view (normal playback or specified CLI behavior).
- **Out of scope (v1):** Editing history order, hiding entries, or streaming art for remote URLs (unless the same doc is updated).
